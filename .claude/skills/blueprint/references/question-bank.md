# 바이브코딩 기획 질문 은행 v0.1

> 사용 규칙 (SKILL.md 와 동일 — 여기 다시 박는다)
> 1. 1단계부터 8단계까지 순서대로 묻는다. 한 번에 한 단계씩.
> 2. 필수 질문에 빈 답이 있으면 다음 단계로 넘어가지 않는다. "모르겠다"도 답이다 — 단, "모르겠다"면 예시를 보여주고 한 번 더 묻는다.
> 3. 조건부 질문은 앞 답이 조건에 해당할 때만 묻는다.
> 4. 각 단계가 끝나면 "이 단계 산출물"을 요약해 사용자에게 확인받은 뒤 진행한다.

---

## 1단계. 아이디어 한 줄

### 필수 질문

**1-1. 만들려는 것을 한 문장으로 말해 주세요. "누구를 위해 / 무엇을 / 핵심 기능 하나" 형식으로.**
- 왜 묻나: "AI 앱" 같은 막연한 문장은 AI도 사람도 헤맨다. 명함 한 줄 소개 수준까지 좁혀야 그다음 질문이 가능하다.
- 예시 답: "1인 자영업자가 / 인스타 카드뉴스를 / 제목만 넣으면 자동으로 만들어 주는 웹."

**1-2. 이것과 비슷한 서비스를 하나만 꼽는다면 무엇인가요? (없으면 "없음")**
- 왜 묻나: 레퍼런스가 있으면 화면·기능 설명이 10배 빨라진다. "○○의 △△ 기능처럼"이 최고의 명세다.
- 예시 답: "미리캔버스요. 근데 템플릿 고르는 과정 없이 제목만 넣으면 바로 나오는 게 달라요."

**1-3. 이걸 왜 지금, 왜 당신이 만들려고 하나요?**
- 왜 묻나: 동기가 명확하면 나중에 기능을 자를 때(스코프컷) 기준이 생긴다. 동기가 흐리면 기능이 무한히 붙는다.
- 예시 답: "제가 매주 카드뉴스 만드는 데 2시간씩 쓰고 있어서요. 내 문제부터 풀고 싶어요."

**1-4. 이게 "성공했다"는 건 어떤 모습인가요? 숫자 하나로 말해 보세요.**
- 왜 묻나: 성공 기준이 있어야 MVP 범위가 정해진다. 기준 없는 프로젝트는 끝이 없다.
- 예시 답: "저 포함 자영업자 10명이 매주 한 번 이상 쓰면 성공."

### 조건부 질문

**1-C1. (조건: 1-1에서 대상이 "모두" 또는 "누구나"라고 답한 경우) 그중에서도 가장 먼저 쓰게 하고 싶은 딱 한 부류는 누구인가요?**
- 왜 묻나: "모두를 위한 서비스"는 아무도 안 쓴다. 첫 사용자 한 부류가 정해져야 화면과 말투가 정해진다.
- 예시 답: "인스타로 홍보하는 동네 카페 사장님."

**1-C2. (조건: 1-2에서 비슷한 서비스가 있다고 답한 경우) 그 서비스를 두고 굳이 새로 만드는 이유, 즉 결정적으로 다른 점 하나는 무엇인가요?**
- 왜 묻나: 차별점이 곧 핵심 기능이다. 차별점이 없으면 그 서비스를 쓰면 된다.
- 예시 답: "기존 건 디자인 툴이라 배워야 하는데, 이건 입력 한 칸만 있으면 돼요."

### 이 단계 산출물
- PRD **"1. 개요(Overview)"** 섹션: 한 줄 정의, 배경·동기, 레퍼런스, 성공 기준 1개.

## 2단계. 타겟·문제

### 필수 질문

**2-1. 이걸 쓸 사람을 실제 인물 한 명처럼 묘사해 주세요. (나이대, 하는 일, 컴퓨터 친숙도)**
- 왜 묻나: 추상적 타겟("2030 여성")으로는 화면 난이도·글씨 크기·말투를 정할 수 없다. 구체적 한 명이 기준이 된다.
- 예시 답: "40대 후반, 동네 카페 운영 7년 차, 스마트폰은 잘 쓰지만 PC는 잘 안 켬."

**2-2. 그 사람이 지금 겪는 불편을 상황 그대로 말해 주세요. (언제, 어디서, 뭘 하다가 막히는지)**
- 왜 묻나: 문제가 구체적이어야 기능이 구체적으로 나온다. "불편하다"가 아니라 장면이 필요하다.
- 예시 답: "일요일 밤에 다음 주 신메뉴 홍보 이미지를 만들려는데, 디자인 툴 켜면 2시간이 사라짐."

**2-3. 그 사람은 지금 이 문제를 어떻게 해결하고 있나요? (수기, 엑셀, 카톡, 다른 앱, 외주 등)**
- 왜 묻나: 현재의 대안이 곧 경쟁 상대다. 대안보다 확실히 편해야 갈아탄다.
- 예시 답: "미리캔버스로 직접 만들거나, 급하면 그냥 사진만 올려요."

**2-4. 그 대안의 어떤 점이 부족해서 이걸 만들려는 건가요?**
- 왜 묻나: 부족한 그 지점이 MVP에서 반드시 해결돼야 할 1순위다.
- 예시 답: "시간이 너무 걸려요. 템플릿 고르다가 지침."

**2-5. 쓰는 사람과 돈 내는 사람이 같은가요? (무료라면 "일단 무료")**
- 왜 묻나: 다르면(예: 직원이 쓰고 사장이 결제) 화면과 권한이 두 벌 필요해진다. 초기에 알아야 구조가 갈린다.
- 예시 답: "같아요. 사장님이 직접 쓰고 직접 결제."

**2-6. 이 문제를 겪는 사람이 대략 몇 명쯤 될까요? 그중 처음 3개월 안에 몇 명이 쓰면 좋겠어요?**
- 왜 묻나: 규모에 따라 나중에 서버·DB 선택이 달라진다. 대부분의 첫 서비스는 "수십~수백 명"이고, 그러면 무료 요금제로 충분하다.
- 예시 답: "전국 카페 사장님이면 수만 명이겠지만, 3개월 목표는 지인 포함 30명."

### 조건부 질문

**2-C1. (조건: 회사·팀에서 쓰는 서비스인 경우) 조직 안에서 등장인물이 몇 명인가요? 각자 뭘 하나요? (예: 직원은 신청, 팀장은 승인, 관리자는 전체 보기)**
- 왜 묻나: 역할이 나뉘면 화면·권한·데이터가 역할 수만큼 갈라진다. 기획 초반에 등장인물을 다 세워야 한다.
- 예시 답: "직원이 회의실을 예약하고, 총무 담당자 1명이 전체 현황을 봐요."

**2-C2. (조건: 타겟이 본인이 아닌 경우) 그 타겟 중 실제로 인터뷰하거나 보여줄 수 있는 사람이 있나요?**
- 왜 묻나: 보여줄 사람이 없으면 상상으로 기획하게 된다. 한 명이라도 있으면 MVP 검증이 가능하다.
- 예시 답: "옆 가게 사장님 두 분한테 보여드리기로 했어요."

### 이 단계 산출물
- PRD **"2. 문제 정의·타겟 사용자"** 섹션: 페르소나 1명, 문제 장면, 현재 대안과 한계, 사용자·결제자 구분, 규모 가늠.

## 3단계. 핵심 기능·스코프컷 (MVP 3개)

### 필수 질문

**3-1. 머릿속에 있는 기능을 전부 쏟아내 보세요. 많아도 됩니다. (나중에 자를 거니까)**
- 왜 묻나: 일단 다 꺼내야 자를 수 있다. 머릿속에만 있으면 개발 중간에 "아 이것도!" 하며 튀어나온다.
- 예시 답: "카드뉴스 자동 생성, 템플릿 선택, 로그인, 내 카드 보관함, 인스타 바로 올리기, 팀 공유, 결제…"

**3-2. 방금 목록에서 "이게 없으면 이 서비스가 아니다" 하는 것 딱 3개만 고르세요. 이게 MVP입니다.**
- 왜 묻나: 첫 버전은 3개면 충분하다. 3개를 완성해 본 사람만 4번째를 만들 자격이 생긴다.
- 예시 답: "① 제목 입력하면 카드뉴스 생성 ② 생성 결과 미리보기 ③ 이미지 다운로드."

**3-3. 고른 3개 각각, "어떻게 되면 완성"인가요? 한 문장씩.**
- 왜 묻나: 완료 기준이 없으면 끝없이 다듬게 된다. AI에게 시킬 때도 이 문장이 검증 기준이 된다.
- 예시 답: "생성: 제목을 넣고 버튼을 누르면 30초 안에 카드 4장이 화면에 뜬다."

**3-4. 나머지 기능은 전부 "나중에" 목록으로 보냅니다. 그중 아까워서 못 버리겠는 것이 있나요?**
- 왜 묻나: 아까운 기능이 스코프를 몰래 불린다. 여기서 명시적으로 "나중에"에 넣고 합의해야 개발이 산으로 안 간다.
- 예시 답: "인스타 자동 업로드가 아까운데… 다운로드만 되면 일단 손으로 올려도 되긴 해요."

**3-5. 반대로, 이번 버전에서 "절대 안 하는 것"을 3개 이상 적어 주세요. (예: 회원가입 없음, 결제 없음, 앱 아니고 웹만)**
- 왜 묻나: 안 하는 것을 문서에 박아야 AI도 사람도 과잉 구현을 안 한다. 하는 것 목록만큼 중요하다.
- 예시 답: "로그인 없음, 결제 없음, 모바일 앱 없음(웹만), 팀 기능 없음."

**3-6. MVP 3개에 순서를 매겨 주세요. 딱 하나만 만들 수 있다면 뭐부터인가요?**
- 왜 묻나: 개발은 한 번에 하나씩 조립한다. 첫 블럭이 정해져야 오늘 뭘 할지가 정해진다.
- 예시 답: "1위 생성, 2위 미리보기, 3위 다운로드. 생성이 되면 나머지는 따라와요."

### 조건부 질문

**3-C1. (조건: 3-2에서 3개로 못 줄이고 4개 이상을 고집하는 경우) 그중 두 개를 겹쳐 놓고 묻겠습니다. 둘 중 하나만 남긴다면 어느 쪽인가요? (남을 때까지 반복)**
- 왜 묻나: 토너먼트로 붙이면 진짜 우선순위가 드러난다. "다 중요해요"는 우선순위가 없다는 뜻이다.
- 예시 답: "'보관함 vs 다운로드'면… 다운로드요. 보관함은 다운받아 두면 되니까."

**3-C2. (조건: MVP에 로그인·결제가 포함된 경우) 로그인·결제 없이도 핵심 가치를 보여줄 수 있는 형태가 있나요?**
- 왜 묻나: 로그인·결제는 만들기도 오래 걸리고 사고도 잦다. 첫 버전에서 뺄 수 있으면 빼는 게 정석이다.
- 예시 답: "생각해 보니 로그인 없이 체험판처럼 열고, 저장이 필요해지면 그때 로그인을 붙여도 되겠네요."

### 이 단계 산출물
- PRD **"3. 기능 요구사항"** 섹션: MVP 3개(각 완료 기준 포함), 우선순위, "나중에" 목록, **Out of Scope(안 하는 것) 목록**.
- 기능명세서의 기능 행 원천 재료.

## 4단계. 유저 플로우 (첫 방문부터 목적 달성까지)

### 필수 질문

**4-1. 처음 온 사람이 주소를 치고 들어오면 제일 먼저 뭐가 보이나요?**
- 왜 묻나: 첫 화면이 플로우의 출발점이다. 여기가 정해져야 나머지 화면이 줄줄이 정해진다.
- 예시 답: "큰 입력창 하나. '카드뉴스 제목을 입력하세요'와 생성 버튼."

**4-2. 그 사람이 목적을 달성할 때까지의 과정을 번호를 붙여 순서대로 말해 주세요. (1. 들어온다 → 2. … → n. 끝)**
- 왜 묻나: 이 번호 목록이 그대로 유저플로우 도식이 되고, 개발 조립 순서가 된다.
- 예시 답: "1. 접속 → 2. 제목 입력 → 3. 생성 버튼 → 4. 30초 대기 → 5. 카드 4장 확인 → 6. 다운로드."

**4-3. 그 과정에 로그인이 필요한 순간이 있나요? 있다면 몇 번째 단계 앞인가요?**
- 왜 묻나: 로그인 위치에 따라 이탈률과 개발 난이도가 확 달라진다. "구경은 자유, 저장부터 로그인"이 보통 정답이다.
- 예시 답: "없어요. MVP는 로그인 없이 다운로드까지."

**4-4. 중간에 갈림길이 있나요? (성공/실패, 결과가 있다/없다, 조건에 따라 다른 화면)**
- 왜 묻나: 갈림길마다 화면과 처리가 하나씩 늘어난다. 미리 세어야 견적이 나온다.
- 예시 답: "생성 실패할 수 있어요. 그러면 '다시 시도' 버튼이 뜨면 좋겠어요."

**4-5. 사용자는 "끝났다, 잘 됐다"를 어떻게 알게 되나요?**
- 왜 묻나: 완료 신호(완료 화면, 알림, 다운로드 파일)가 없으면 사용자는 불안해서 버튼을 또 누른다.
- 예시 답: "다운로드 버튼을 누르면 파일이 저장되고 '저장 완료' 메시지가 뜬다."

**4-6. 두 번째 방문한 사람의 흐름은 첫 방문과 다른가요? (이어하기, 지난 기록 보기 등)**
- 왜 묻나: 재방문 흐름에서 보통 "저장·목록·계정" 요구가 튀어나온다. MVP에 넣을지 여기서 판단한다.
- 예시 답: "같아요. 기록 보관은 '나중에' 목록에 있어요."

**4-7. 운영자(당신)의 흐름도 필요한가요? 사용자 몰래 뒤에서 뭘 확인하거나 처리해야 하나요?**
- 왜 묻나: 관리자 화면은 잊기 쉬운 숨은 스코프다. 초기엔 DB 대시보드를 직접 보는 것으로 대체 가능한지 판단한다.
- 예시 답: "몇 명이 생성했는지만 보고 싶어요. 처음엔 DB 화면을 직접 봐도 될 것 같아요."

### 조건부 질문

**4-C1. (조건: 결제가 있는 경우) 결제는 흐름의 몇 번째 단계에 들어가나요? 결제 전과 후에 사용자가 볼 수 있는 것이 어떻게 다른가요?**
- 왜 묻나: 결제 위치가 화면 구조와 권한 구조를 결정한다. 결제는 실수 비용이 커서 흐름을 따로 그려야 한다.
- 예시 답: "월 10장까지 무료, 11장째부터 결제 안내 페이지로 이동."

**4-C2. (조건: 승인·확인 절차가 있는 경우 — 예약, 신청, 주문 등) 신청 후 승인까지 사용자는 뭘 보고 기다리나요? (대기 중 상태 표시)**
- 왜 묻나: "대기 중"이라는 중간 상태가 생기면 데이터에 상태 항목이 필요해진다. 6단계 설계에 직결된다.
- 예시 답: "'예약 확인 중' 표시가 보이고, 승인되면 '확정'으로 바뀐다."

**4-C3. (조건: 알림이 필요한 경우) 사용자가 사이트를 닫은 뒤에도 알려줘야 하는 일이 있나요? (이메일, 카카오톡 등)**
- 왜 묻나: 화면 밖 알림은 외부 연동(7단계)이 필요하다. 여기서 표시해 두고 7단계에서 매핑한다.
- 예시 답: "예약이 확정되면 이메일 한 통은 가면 좋겠어요."

### 이 단계 산출물
- PRD **"4. 유저 플로우"** 섹션: 번호 매긴 메인 플로우 + 갈림길(분기) + 재방문·운영자 플로우. 화살표 도식 1장으로 그릴 수 있는 상태.

## 5단계. 화면 목록·와이어프레임

### 필수 질문

**5-1. 4단계 플로우를 따라가며 필요한 화면(페이지)을 전부 나열해 주세요.**
- 왜 묻나: 화면 목록 = 개발 분량이다. 화면 수를 세는 순간 일의 크기가 눈에 보인다.
- 예시 답: "① 메인(입력) ② 생성 중(로딩) ③ 결과(미리보기+다운로드) ④ 실패 안내. 총 4개."

**5-2. 화면 하나마다, 그 화면에서 사용자가 하는 일을 딱 한 가지로 말해 주세요.**
- 왜 묻나: 한 화면에 일이 두 개면 화면을 쪼개야 한다는 신호다. "한 화면 = 한 목적"이 좋은 설계다.
- 예시 답: "메인 화면에서 하는 일은 '제목을 입력하고 생성 버튼을 누른다', 그거 하나."

**5-3. 각 화면을 위에서 아래로 스크롤한다고 상상하고, 순서대로 뭐가 보이는지 말해 주세요. (로고, 제목, 입력칸, 버튼, 목록…)**
- 왜 묻나: 이 답이 그대로 와이어프레임이고, 기능명세서의 "No | 구분 | 기능 설명" 행이 된다.
- 예시 답: "결과 화면: 맨 위 로고 → '완성됐어요' 문구 → 카드 4장 나란히 → 다운로드 버튼 → 다시 만들기 링크."

**5-4. 각 화면의 버튼·링크를 누르면 각각 어디로 가나요?**
- 왜 묻나: 화면 사이의 연결선이 빠지면 "막다른 화면"이 생긴다. 모든 버튼의 목적지를 확인한다.
- 예시 답: "다운로드 → 파일 저장(화면 유지), 다시 만들기 → 메인으로."

**5-5. 데이터가 하나도 없을 때 각 화면은 어떻게 보이나요? (첫 가입 직후, 검색 결과 0건 등)**
- 왜 묻나: 빈 상태(empty state)는 항상 잊힌다. 빈 화면에 안내 문구가 없으면 사용자는 고장 난 줄 안다.
- 예시 답: "보관함이 비었으면 '아직 만든 카드가 없어요. 첫 카드를 만들어 보세요' + 만들기 버튼."

**5-6. 주로 스마트폰으로 보나요, PC로 보나요? 둘 다라면 어느 쪽이 우선인가요?**
- 왜 묻나: 우선 기기에 따라 레이아웃 설계(1열 vs 3열)와 확인 방법이 달라진다.
- 예시 답: "사장님들은 폰으로 봐요. 모바일 우선, PC는 보이기만 하면 됨."

### 조건부 질문

**5-C1. (조건: 목록 화면이 있는 경우) 목록의 한 줄(한 칸)에는 뭐가 보여야 하나요? 한 화면에 몇 개까지 보여주고, 더 많으면 어떻게 하나요? (더보기, 페이지 넘김)**
- 왜 묻나: 목록의 한 줄 구성이 데이터 항목(6단계 컬럼)과 직결된다. 개수 처리를 안 정하면 천 개가 한 화면에 쏟아진다.
- 예시 답: "예약 목록 한 줄: 날짜, 회의실 이름, 상태. 최근 20개, 아래 '더보기' 버튼."

**5-C2. (조건: 입력 화면이 있는 경우) 입력 칸을 전부 나열하고, 각각 필수인지 선택인지 표시해 주세요. 잘못 입력하면 뭐라고 알려주나요?**
- 왜 묻나: 입력 항목 = 저장 항목이다. 필수·선택과 오류 문구를 정해야 폼이 완성된다.
- 예시 답: "제목(필수, 30자 이내), 부제(선택). 비우고 누르면 '제목을 입력해 주세요'."

**5-C3. (조건: 로그인이 있는 경우) 로그인한 사람과 안 한 사람에게 화면이 어떻게 다르게 보이나요?**
- 왜 묻나: 상태별 화면 차이는 은근히 큰 작업량이다. 미리 정해야 헤더·메뉴 설계가 된다.
- 예시 답: "안 했으면 우상단에 '로그인' 버튼, 했으면 내 이름과 '보관함' 메뉴."

**5-C4. (조건: 관리자 화면이 필요한 경우) 관리자 화면에는 어떤 목록·버튼이 필요한가요? 일반 사용자가 이 주소를 알면 들어올 수 있으면 안 되죠?**
- 왜 묻나: 관리자 화면은 별도 화면 + 별도 권한이다. 접근 차단을 여기서 명시해야 8단계 보안으로 이어진다.
- 예시 답: "전체 예약 목록과 '승인/거절' 버튼. 관리자 계정으로만 접근."

### 이 단계 산출물
- PRD **"5. 화면 정의"** 섹션: 화면 목록표(화면명·목적·구성요소·이동 관계).
- 기능명세서(02-spec.md)의 화면별 행 + 와이어프레임(04-wireframe.html)을 그릴 수 있는 상태.

## 6단계. 데이터 모델 (DB 구조)

> 이 단계가 가장 길다. 전부 답해야 넘어간다. 답이 곧 표(테이블) 설계도가 된다.

### 필수 질문

**6-1. 이 서비스가 기억해야 할 것(저장할 것)을 명사로 전부 나열해 주세요. 껐다 켜도 남아 있어야 하는 것들입니다.**
- 왜 묻나: 명사 하나가 대체로 표(테이블) 하나가 된다. 이 목록이 DB 설계의 출발점이다.
- 예시 답: "회원, 예약, 회의실. 세 가지."

**6-2. 그 명사 하나하나에 어떤 정보(항목)가 붙나요? 엑셀 표의 열 제목을 짓는다고 생각하세요.**
- 왜 묻나: 항목(컬럼) 목록이 곧 표의 구조다. 입력 화면(5-C2)의 칸과 일치하는지 여기서 맞춰 본다.
- 예시 답: "예약: 누가, 어느 회의실, 날짜, 시작 시각, 끝 시각, 상태(대기/확정/취소)."

**6-3. 각 항목은 어떤 종류인가요? 글자 / 숫자 / 날짜·시각 / 예·아니오 중에서.**
- 왜 묻나: 종류(타입)를 정해야 계산·정렬·검색이 제대로 된다. 날짜를 글자로 저장하면 나중에 정렬이 다 깨진다.
- 예시 답: "날짜는 날짜, 인원수는 숫자, 상태는 글자, '반복 예약 여부'는 예·아니오."

**6-4. 각 항목은 필수인가요, 비워도 되나요?**
- 왜 묻나: 필수 여부가 입력 화면 검증과 DB 제약을 결정한다. 나중에 바꾸면 기존 데이터가 걸린다.
- 예시 답: "날짜·시각·회의실은 필수, 메모는 선택."

**6-5. 각 데이터는 누구의 것인가요? 특정 회원의 것인가요, 서비스 공용인가요?**
- 왜 묻나: 소유자가 정해져야 "누구 것" 꼬리표(회원 번호 연결)와 권한(6-9)이 정해진다.
- 예시 답: "예약은 예약한 회원의 것. 회의실 목록은 공용(관리자만 수정)."

**6-6. "하나가 여러 개를 가지는" 쌍을 찾아 주세요. (예: 회원 1명이 예약 여러 건 = 1:N)**
- 왜 묻나: 1:N 관계마다 꼬리표(연결 번호)가 하나 필요하다. 택배 송장의 '받는 사람'처럼.
- 예시 답: "회원 1명이 예약 여러 건. 회의실 1개에도 예약 여러 건."

**6-7. "서로가 서로를 여러 개 가지는" 쌍이 있나요? (예: 학생 여럿이 강의 여럿을 수강 = N:M)**
- 왜 묻나: N:M은 짝을 적는 연결표가 하나 더 필요하다. 있는지 없는지만 확인하면 만드는 건 AI가 한다.
- 예시 답: "없어요. (있는 예: 게시글 여러 개에 태그 여러 개.)"

**6-8. 같은 값이 두 번 있으면 안 되는 항목이 있나요? (이메일, 아이디, 쿠폰 코드 등)**
- 왜 묻나: 중복 금지(유니크)를 걸어 두면 같은 이메일로 두 번 가입하는 사고를 DB가 막아 준다.
- 예시 답: "회원 이메일은 중복 금지. 같은 회의실·같은 시간대 예약도 중복 금지."

**6-9. 각 표를 누가 볼 수 있나요? 본인만 / 로그인한 사람 모두 / 아무나 / 관리자만 중에서 표마다 정해 주세요.**
- 왜 묻나: 이 답이 그대로 열람 권한 규칙(RLS: 데이터 창고 문 앞의 출입 규칙)이 된다. 정하지 않으면 남의 데이터가 보이는 사고가 난다.
- 예시 답: "예약은 본인 것만 + 관리자는 전체. 회의실 목록은 로그인한 사람 모두."

**6-10. 각 표에 누가 새로 쓰고, 누가 고칠 수 있나요? 특히 "아무나 수정·삭제"가 되면 안 되는 것은?**
- 왜 묻나: 읽기와 쓰기는 권한이 다르다. 사용자 데이터에 공개 수정·삭제를 열면 누구든 남의 데이터를 지울 수 있다 — 절대 금지 항목이다.
- 예시 답: "예약은 본인만 만들고 취소 가능. 회의실 추가·수정은 관리자만."

**6-11. 데이터를 지우면 어떻게 할까요? 진짜로 지울까요, 아니면 '지워짐' 표시만 하고 남겨둘까요? 연결된 데이터(그 회원의 예약들)는요?**
- 왜 묻나: 진짜 삭제는 복구가 안 된다. 기록·복구가 필요하면 표시만 하는 방식(soft delete)이 안전하다. 연결 데이터 처리를 안 정하면 유령 데이터가 남는다.
- 예시 답: "예약 취소는 '취소됨' 표시로 남기고, 회원 탈퇴 시 개인정보만 지우고 예약 기록은 익명으로 보관."

**6-12. 사용자는 무엇으로 데이터를 찾나요? (이름으로 검색, 날짜로 검색 등)**
- 왜 묻나: 검색 기준 항목은 빨리 찾도록 준비(인덱스)해 둬야 한다. 데이터가 쌓인 뒤에 느려지고 나서 고치면 늦다.
- 예시 답: "날짜로 그날 예약을 찾고, 관리자는 회원 이름으로도 찾아요."

**6-13. 목록의 기본 정렬 순서는요? (최신순, 이름순, 가격순 등)**
- 왜 묻나: 정렬 기준도 검색처럼 미리 준비할 항목이다. 화면(5-C1)의 목록 설계와 일치해야 한다.
- 예시 답: "예약 목록은 다가오는 날짜순, 관리자 화면은 신청 최신순."

**6-14. 저장하는 것 중 개인정보가 있나요? (이름, 전화번호, 주소, 생년월일 등)**
- 왜 묻나: 개인정보는 최소한만 수집하고, 보여줄 때·지울 때 규칙이 따로 필요하다. 유출되면 법적 책임까지 가는 데이터다.
- 예시 답: "이름과 이메일만. 전화번호는 안 받기로 했어요."

**6-15. 서비스를 여는 첫날, 미리 넣어 둬야 하는 데이터가 있나요? (카테고리 목록, 회의실 목록, 관리자 계정 등)**
- 왜 묻나: 초기 데이터(시드)가 없으면 첫 화면이 텅 빈다. 누가 언제 넣을지 정해야 오픈이 된다.
- 예시 답: "회의실 5개 목록과 관리자 계정 1개는 미리 넣어야 해요."

**6-16. 각 표가 1년 뒤 몇 건쯤 될까요? 대충이라도. (수백 / 수만 / 수백만)**
- 왜 묻나: 규모에 따라 설계와 요금제가 달라진다. 대부분의 첫 서비스는 수만 건 이하라 무료 DB로 충분하다 — 과잉 설계를 막는 질문이기도 하다.
- 예시 답: "회원 100명, 예약은 하루 20건이면 1년에 7천 건쯤."

### 조건부 질문

**6-C1. (조건: 6-7에서 N:M이 있다고 답한 경우) 그 연결 자체에 붙는 정보가 있나요? (예: 수강 시작일, 태그 단 날짜)**
- 왜 묻나: 연결표에도 항목이 붙을 수 있다. 붙는다면 연결표가 어엿한 표 하나로 승격된다.
- 예시 답: "수강이면 '신청일'과 '수료 여부'가 연결에 붙어요."

**6-C2. (조건: 4-C2처럼 상태(대기/확정/취소 등)가 있는 경우) 상태의 전체 목록과, 어떤 상태에서 어떤 상태로만 바뀔 수 있는지 말해 주세요.**
- 왜 묻나: 상태 흐름을 안 정하면 "취소된 예약이 다시 확정되는" 버그가 생긴다. 상태표는 종이에 화살표로 그릴 수 있어야 한다.
- 예시 답: "대기 → 확정 또는 취소. 확정 → 취소만 가능. 취소는 끝."

**6-C3. (조건: 6-11에서 '표시만 하고 남긴다'를 고른 경우) 지워진 데이터를 복구해 달라는 요청이 오면 누가, 어떻게 복구하나요?**
- 왜 묻나: 남겨두는 이유가 복구라면 복구 절차도 있어야 한다. 없다면 관리자 화면 요구가 하나 늘어난다.
- 예시 답: "당분간은 저(운영자)가 DB 화면에서 직접 표시를 되돌리는 걸로."

**6-C4. (조건: 6-14에서 개인정보가 있다고 답한 경우) 회원이 탈퇴하면 그 개인정보는 언제, 어떻게 지우나요?**
- 왜 묻나: 탈퇴 시 파기는 법적 의무 영역이다. "탈퇴 즉시 삭제"인지 "30일 보관 후 삭제"인지 정책이 필요하다.
- 예시 답: "탈퇴 즉시 이름·이메일 삭제, 예약 기록은 '탈퇴 회원'으로 익명 처리."

### 이 단계 산출물
- PRD **"6. 데이터 모델"** 섹션 + `05-db.md`: 표 목록, 표별 항목(이름·종류·필수·중복금지), 관계 도식(1:N 꼬리표·N:M 연결표), 표별 읽기·쓰기 권한표, 삭제 정책, 검색·정렬 기준, 초기 데이터 목록.
- 그대로 DB 테이블 + 출입 규칙(RLS) 생성 지시문으로 쓸 수 있는 상태. (모든 표에 만든 시각·고친 시각 자동 기록은 기본 포함)

## 7단계. 외부 연동·키 (.env 매핑)

> 외부 서비스를 하나 붙일 때마다 "가입 → 키 발급 → .env에 키 저장" 작업이 하나씩 늘어난다. 여기서 전부 세어 둔다.
> **.env 파일(비밀 열쇠 보관 파일)은 절대 인터넷(깃허브)에 올리지 않는다.**
>
> **기본 스택 (묻지 않고 자동 포함)**: 저장·로그인 = **Supabase**, 배포 = **Vercel**. 밀키트의 기본 재료라 인터뷰 없이 연동 목록에 기본으로 올린다 — 단계 시작 때 "이 둘은 기본으로 깔립니다" 한 줄만 고지한다.
> **Cloudflare 는 조건부**: 7-7(도메인)·7-8(이미지 저장량) 답에 따라 넣거나 뺀다. 판정 규칙 — **7-7=내 도메인** 또는 **7-8=많이** 중 하나라도 해당하면 포함(사유 명시), 둘 다 아니면 산출물에 아예 등장시키지 않는다.

### 필수 질문

**7-1. 로그인이 필요한가요? 필요하다면 어떤 방식으로? — 없음 / 이메일+비밀번호 / 소셜 로그인(구글·카카오·네이버·애플)**
- 왜 묻나: 로그인 방식에 따라 준비물(외부 서비스 등록·키)이 달라진다. 소셜 하나당 등록 작업이 하나씩 늘어난다.
- 예시 답: "구글 로그인 하나만. 카카오는 나중에."

| 선택 | 플랫폼 | 필요한 .env 키 |
|---|---|---|
| 어떤 로그인이든 | Supabase Auth | `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` |
| 구글 로그인 | Google Cloud Console | `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` (Supabase 대시보드에 등록) |
| 카카오 로그인 | Kakao Developers | `KAKAO_CLIENT_ID`, `KAKAO_CLIENT_SECRET` (Supabase 대시보드에 등록) |
| 네이버 로그인 | Naver Developers | `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET` |

**7-2. 돈을 받나요? 받는다면 — 일회성 결제 / 정기 구독 / 아직 안 받음**
- 왜 묻나: 결제는 연동 중 가장 조심할 영역이다. 반드시 테스트 키(가짜 돈)로 먼저 만들고, 실제 키 전환은 맨 마지막에 한다.
- 예시 답: "MVP는 무료. 구독은 나중에."

| 선택 | 플랫폼 | 필요한 .env 키 |
|---|---|---|
| 국내 카드결제 | 토스페이먼츠 | `NEXT_PUBLIC_TOSS_CLIENT_KEY`, `TOSS_SECRET_KEY` (테스트 키부터) |
| 국내 통합 | 포트원(구 아임포트) | `PORTONE_API_KEY`, `PORTONE_API_SECRET` |
| 해외·구독 | Stripe | `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` |

**7-3. 사용자에게 이메일이나 알림을 보내야 하나요? 언제, 무엇을?**
- 왜 묻나: "확정되면 알려주기" 한 줄이 외부 서비스 하나를 부른다. 보낼 순간과 내용을 정해야 견적이 나온다.
- 예시 답: "예약 확정 때 이메일 한 통. 카톡 알림은 나중에."

| 선택 | 플랫폼 | 필요한 .env 키 |
|---|---|---|
| 이메일 | Resend | `RESEND_API_KEY` |
| 카카오 알림톡·SMS | 솔라피(Solapi) | `SOLAPI_API_KEY`, `SOLAPI_API_SECRET` |

**7-4. AI 기능이 들어가나요? (글 생성, 요약, 챗봇, 이미지 등) 들어간다면 정확히 뭘 시키나요?**
- 왜 묻나: AI 호출은 건당 돈이 나간다. 뭘 시킬지 문장으로 정해야 키 발급과 비용 예측이 된다.
- 예시 답: "제목을 주면 카드뉴스 문구 4개를 만들어 주는 것. 그거 하나."

| 선택 | 플랫폼 | 필요한 .env 키 |
|---|---|---|
| Claude | Anthropic | `ANTHROPIC_API_KEY` |
| GPT | OpenAI | `OPENAI_API_KEY` |
| Gemini | Google AI Studio | `GEMINI_API_KEY` |

**7-5. 사용자가 파일(사진·문서)을 올리나요? 올린다면 종류·최대 크기·1인당 개수는?**
- 왜 묻나: 파일 저장은 DB가 아니라 별도 창고(스토리지)를 쓴다. 크기 제한을 안 정하면 용량 요금 폭탄이 온다.
- 예시 답: "가게 로고 이미지 1장, 5MB 이하."

| 선택 | 플랫폼 | 필요한 .env 키 |
|---|---|---|
| 기본(권장) | Supabase Storage | Supabase 키 재사용 (추가 키 없음) |
| 이미지 변환까지 | Cloudinary | `CLOUDINARY_URL` |

**7-6. 그 밖에 빌려 쓸 외부 기능이 있나요? (지도, 주소 검색, 캘린더 연동, 문자 인증, 엑셀 내보내기 등)**
- 왜 묻나: 숨은 연동을 지금 다 꺼내야 한다. 개발 중간에 "지도도 필요한데"가 나오면 일정이 밀린다.
- 예시 답: "가게 위치 지도 하나 넣고 싶어요."

| 선택 | 플랫폼 | 필요한 .env 키 |
|---|---|---|
| 지도 | Kakao Maps | `NEXT_PUBLIC_KAKAO_MAP_APP_KEY` |
| 주소 검색 | 다음 우편번호 | 키 불필요 (무료 스크립트) |
| 문자 인증 | 솔라피 등 | `SOLAPI_API_KEY`, `SOLAPI_API_SECRET` |

**7-7. 서비스 주소는 어떻게 열 건가요? — 일단 무료 주소(내이름.vercel.app) / 내 도메인이 이미 있다 / 도메인을 새로 살 생각**
- 왜 묻나: 내 도메인(mycafe.com)을 붙이는 순간 "주소를 서버로 안내하는 전화번호부(DNS)" 관리가 필요해진다. 무료 주소면 이 작업이 통째로 없다. 도메인을 쓴다면 Cloudflare 로 DNS 를 관리하는 게 기본기(무료·빠름)다.
- 예시 답: "일단 vercel.app 무료 주소로 열고, 반응 오면 도메인 살게요."

| 선택 | 플랫폼 | 필요한 것 |
|---|---|---|
| 무료 주소 | Vercel 기본 도메인 | 없음 (Cloudflare 불포함) |
| 내 도메인 | **Cloudflare DNS** | 계정 + 도메인 네임서버 변경 (대시보드 작업 — .env 키 불필요) |

**7-8. 사진·이미지 같은 파일을 얼마나 저장하게 되나요? — 거의 없음 / 가끔(로고·프로필 수준) / 많이(사용자들이 계속 올림)** (7-5에서 "안 올린다"고 했으면 "거의 없음"으로 자동 기록)
- 왜 묻나: 소량이면 Supabase 창고(무료 1GB)로 충분하다. 사용자들이 이미지를 계속 올리는 서비스면 저장보다 **내려받는 전송량 요금**이 커지는데, Cloudflare R2 는 전송료가 무료라 이미지 창고로 유리하다.
- 예시 답: "가게 로고 한 장 정도예요." → Supabase Storage 로 충분, Cloudflare 불포함.

| 선택 | 플랫폼 | 필요한 .env 키 |
|---|---|---|
| 거의 없음·가끔 | Supabase Storage (기본) | Supabase 키 재사용 (추가 키 없음) |
| 많이 (이미지 위주 서비스) | **Cloudflare R2** | `CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_API_TOKEN` |

### 조건부 질문

**7-C1. (조건: 결제를 받는 경우) 환불은 어떻게 하나요? 자동인가요, 요청받아 수동인가요? 환불 가능 기간은?**
- 왜 묻나: 환불 정책 없이 결제를 열면 분쟁이 생긴다. 초기엔 "요청 시 운영자가 수동 환불"이 가장 안전하다.
- 예시 답: "결제 후 7일 이내 요청 시 제가 직접 환불 처리."

**7-C2. (조건: AI를 쓰는 경우) 한 사람이 하루에 몇 번까지 쓰게 할까요? 무제한이면 비용을 감당할 수 있나요?**
- 왜 묻나: 호출 제한이 없으면 한 명이 요금을 다 태울 수 있다. 사용량 제한은 설계 단계에서 넣어야 한다.
- 예시 답: "무료 사용자는 하루 3회, 그 이상은 나중에 유료로."

**7-C3. (조건: 외부 연동이 하나라도 있는 경우) 각 서비스의 계정을 지금 가지고 있나요? 결제 카드 등록이 필요한 곳은 어디인가요?**
- 왜 묻나: 키 발급은 개발 전 준비물이다. 계정·카드 문제로 개발 당일 막히는 경우가 가장 흔하다.
- 예시 답: "구글 계정은 있고, Anthropic은 카드 등록이 필요하다고 들었어요."

### 이 단계 산출물
- PRD **"7. 외부 연동"** 섹션: **기본 스택 줄**(Supabase · Vercel [· Cloudflare + 포함 사유(DNS/R2)]) + 연동 목록표(기능 - 플랫폼(서비스 실명) - 발급처 - .env 키 이름). 붙이는 외부 서비스는 전부 **이름을 명시**한다("결제 모듈" 금지 → "토스페이먼츠").
- `.env.example` 대조: 필요한 키가 없으면 추가. 개발 시작 전 발급 체크리스트. Cloudflare 미포함 판정이면 관련 키를 체크리스트에 넣지 않는다.

## 8단계. 비기능 (권한·보안·실패·운영)

### 필수 질문

**8-1. 사용자 등급이 몇 단계인가요? (예: 비회원 / 일반 회원 / 관리자) 등급마다 할 수 있는 일을 한 줄씩.**
- 왜 묻나: 등급표가 있어야 6단계 권한 규칙과 5단계 화면 차이가 최종 검증된다. 관리자를 잊으면 운영을 못 한다.
- 예시 답: "비회원: 구경만 / 회원: 예약 생성·취소 / 관리자(저 1명): 전체 보기·승인."

**8-2. 관리자는 누구이고 몇 명인가요? 관리자 계정은 어떻게 만드나요?**
- 왜 묻나: "관리자 승격" 경로를 안 정하면, 아무나 관리자가 되는 구멍이 생기거나 아무도 못 되는 잠금이 생긴다.
- 예시 답: "저 혼자. 제 계정을 DB에서 직접 관리자로 표시해 두는 방식."

**8-3. 이 서비스에서 유출·조작되면 가장 큰일 나는 데이터는 무엇인가요?**
- 왜 묻나: 지킬 것의 우선순위를 정해야 보안에 힘을 어디에 쓸지 정해진다. 전부 지키려다 전부 허술해진다.
- 예시 답: "회원 이메일 목록이요. 그거 유출되면 신뢰가 끝나요."

**8-4. 저장·결제·생성이 실패하면 사용자에게 뭐라고 보여줄까요? 실패한 데이터는 어떻게 되나요?**
- 왜 묻나: 실패는 반드시 일어난다. 실패 화면과 문구를 안 정하면 사용자는 하얀 화면 앞에서 이탈한다.
- 예시 답: "'잠시 문제가 생겼어요. 다시 시도해 주세요' + 다시 시도 버튼. 입력한 내용은 남아 있게."

**8-5. 빌려 쓰는 외부 서비스(AI, 결제 등)가 먹통이 되면 서비스는 어떻게 되나요? 그래도 되나요?**
- 왜 묻나: 외부 장애는 내 잘못이 아니어도 내 서비스가 멈춘다. "그동안 이 기능만 잠시 꺼진다"를 미리 정하면 당황하지 않는다.
- 예시 답: "AI가 죽으면 생성만 안 되고, '지금 생성이 밀려 있어요' 안내가 뜨면 돼요."

**8-6. 문의·버그 신고는 어디로 받나요? 응답은 누가 하나요?**
- 왜 묻나: 오픈하면 반드시 연락이 온다. 채널이 없으면 사용자는 조용히 떠난다. 채널 링크는 화면 어딘가에 있어야 한다(5단계에 반영).
- 예시 답: "카카오 채널 하나 만들어서 화면 하단에 링크. 답은 제가 해요."

**8-7. 데이터가 날아가면 어디까지 잃어도 견딜 수 있나요? 백업은 얼마나 자주 할까요?**
- 왜 묻나: 백업 주기는 "잃어도 되는 양"으로 정한다. Supabase 무료 요금제의 자동 백업 범위도 확인 대상이다.
- 예시 답: "하루치까지는 견딜 수 있어요. 매일 자동 백업이면 충분."

### 조건부 질문

**8-C1. (조건: 결제 또는 개인정보가 있는 경우) 이용약관·개인정보처리방침 페이지를 준비했나요?**
- 왜 묻나: 개인정보를 받는 순간 처리방침 게시는 의무다. 화면 목록(5단계)에 페이지 2개가 추가된다.
- 예시 답: "아직요. 표준 템플릿으로 만들어서 하단에 링크할게요."

**8-C2. (조건: 동시 사용자가 몰릴 수 있는 경우 — 선착순 신청, 이벤트 등) 몇 명이 같은 순간에 몰릴 수 있나요? 같은 것을 두 명이 동시에 잡으면 누가 이기나요?**
- 왜 묻나: 선착순은 동시 접근 처리(같은 회의실을 동시에 예약)가 핵심 난이도다. 규칙을 정해야 AI가 안전하게 구현한다.
- 예시 답: "공지 직후 30명쯤. 먼저 저장된 사람이 이기고, 늦은 사람에겐 '방금 마감됐어요'."

**8-C3. (조건: 성장 계획이 있는 경우) 3개월 뒤 추가하고 싶은 기능("나중에" 목록)이 지금 구조를 뒤집을 만한 것인가요? (예: 지금은 1인용인데 나중에 팀 기능)**
- 왜 묻나: 다 대비할 필요는 없지만, "1인용 → 팀용" 같은 전환은 데이터 구조가 뒤집힌다. 그런 것 하나만 미리 표시해 둔다.
- 예시 답: "팀 공유가 예정돼 있어요. 그럼 데이터에 '팀' 개념 자리만 미리 잡아 두자는 거군요."

### 이 단계 산출물
