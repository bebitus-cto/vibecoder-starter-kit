# blueprint — 기획 인터뷰 스킬

> 목적: 사용자(비개발자·입문 바이브코더)의 아이디어를 **아주 자세한 질문**으로 파고들어,
> 개발을 바로 시작할 수 있는 설계 문서 5종을 만든다. **질문 없이 산출물부터 쓰는 것 금지.**
>
> **설계 승인 게이트**: 이 인터뷰가 끝나 산출물이 승인되기 전에는 코드를 쓰지 않는다. 아무리 간단해 보여도.

## 산출물 (전부 `docs/plan/` 에 생성)

| 파일 | 내용 |
|------|------|
| `docs/plan/01-prd.md` | PRD — 개요·문제/타겟·기능(MVP+안하는것)·비기능 |
| `docs/plan/02-spec.md` | 기능명세서 표 — 화면별 No/구분/기능/설명/우선순위 |
| `docs/plan/03-userflow.md` | 유저플로우 — mermaid 도식 + 갈림길 |
| `docs/plan/04-wireframe.html` | 로파이 와이어프레임 — 회색 박스, 화면별 |
| `docs/plan/05-db.md` | DB 설계 — mermaid erDiagram + 표별 권한/삭제정책 + SQL 초안 |
| (갱신) `plan.md` | 프로젝트 루트 기획서 — 한 문장 소개·데이터·블럭 체크리스트·스코프컷 |
| (대조) `.env.example` | 7단계에서 나온 연동에 필요한 키가 다 있는지 확인, 없으면 추가 |

## 절차 (반드시 이 순서)

### 0. 시작
- 사용자가 아이디어를 줬으면 그것을 1단계 답의 초안으로 삼는다. 없으면 1단계부터 묻는다.
- 선언하라: "8단계 질문으로 설계도를 만든다. 한 단계씩 간다. 대답이 곧 문서가 된다."
- 선언한 그 턴에서 바로 1단계 첫 질문을 AskUserQuestion 카드로 던진다. 설명만 하고 카드 없이 턴을 끝내지 마라.

### 1~8. 인터뷰 (질문 은행 = `references/question-bank.md`)
단계: ① 아이디어 한 줄 ② 타겟·문제 ③ 핵심기능·스코프컷(MVP 3개) ④ 유저플로우 ⑤ 화면·와이어프레임 ⑥ 데이터 모델(DB) ⑦ 외부 연동·키 — **Supabase·Vercel 은 기본 스택(안 묻고 자동 포함), Cloudflare 는 도메인(DNS)·이미지 저장량(R2) 문답으로 조건부 포함** ⑧ 비기능(권한·보안·실패·운영)

**인터뷰 규칙 (강제):**
1. **한 번에 한 단계.** 단계 안에서도 질문을 2~4개씩 끊어 묻는다 (한 번에 16개 던지지 않는다).
2. **필수 질문에 빈 답이 있으면 다음 단계로 못 넘어간다.** "모르겠다"도 답이다 — 단, 그 경우 질문 은행의 예시 답을 보여주고 한 번 더 묻는다. 그래도 모르면 "미정"으로 기록하고 문서에 `[미정]` 표시.
3. **조건부 질문**은 앞 답이 조건에 해당할 때만 묻는다 (은행에 조건 명시돼 있음).
4. **단계가 끝나면 그 단계 산출물을 3~6줄로 요약해 보여주고 "맞습니까?" 확인** 후 다음 단계로.
5. **질문은 AskUserQuestion 도구로 제시한다(선택 카드 UI).** 두 가지가 핵심이다:
   - **선택지는 미리 정해두지 않고 추론 생성한다.** 사용자의 아이디어·업종·앞 단계 답변 맥락에서 그럴듯한 선택지 2~4개를 그때그때 만든다. 질문 은행의 예시 선택지는 참고일 뿐, 그대로 박지 마라. (예: 카드뉴스 앱이면 "누구를 위해?"에 [동네 자영업자 / 인플루언서 / 직장인 부업 / 비영리 단체] 처럼 그 맥락에 맞게 생성.)
   - **기타(자유 입력)는 항상 열려 있다.** AskUserQuestion 이 자동으로 "기타" 를 제공하니, 선택지에 없는 답을 사용자가 직접 적을 수 있다. 그래서 서술형에 가까운 질문(타겟·문제 등)도 선택지 몇 개 + 기타 형태로 물을 수 있다.
   - **예외 없음 — 모든 질문을 AskUserQuestion으로 묻는다. 텍스트로 질문만 적고 턴을 끝내는 것 금지.** (텍스트 질문은 사용자가 안 답하고 지나칠 수 있지만, 카드는 답하기 전엔 진행이 안 된다.) 1-1 아이디어 한 줄 같은 개방형도 카드로 묻는다: 선택지에 대표 유형 3~4개(예: 예약·신청 받기 / 기록·관리 도구 / 홍보 페이지 / 판매·결제)를 놓고, 질문 본문에 "떠오르는 문장이 있으면 '기타'에 그대로 적어주세요"를 반드시 넣는다. 여러 항목을 한 화면에 물을 땐 AskUserQuestion 한 번에 질문 2~4개를 묶어도 된다.
   - **특히 ⑥ 데이터 모델은 전 질문을 AskUserQuestion 으로만 진행한다(텍스트 서술 금지) — 은행의 "6단계 카드 흐름"을 따른다.**
6. 전문용어가 나오면 그 자리에서 한 줄 비유로 풀어라 (예: RLS = 창고 문 앞의 출입 규칙).
7. 사용자가 "빨리 넘어가자"고 해도 ⑥ 데이터 모델의 필수 질문은 생략 불가 — "여기서 아낀 10분이 나중에 10시간이 된다"고 말하고 진행한다.
8. **펜을 돌려줘라.** 너는 사용자 대신 결정하지 않는다. "X를 Y로 이해했는데 맞습니까?"는 되고, 사용자 대신 기능을 발명하거나 MVP를 네가 잘라주는 것은 안 된다. 네가 저작하기 시작했다고 느끼면 멈추고 질문으로 돌아가라. (학습 목적: AI가 다 정해주면 사용자가 자기 프로젝트를 이해 못 한다.)
9. **추천 답을 1순위로.** 앞 단계 답·업종 맥락에서 답을 추정할 수 있으면 첫 선택지에 "(추천)"을 붙이고 근거를 한 줄 단다. 사용자는 "맞아"만 눌러도 굴러가야 한다 — 모르는 사람일수록 백지 질문이 무섭다. (단 추천은 제안이지 결정이 아니다 — 규칙 8과 충돌하면 8이 이긴다.)
10. **묻기 전에 먼저 찾아라.** 기존 plan.md·docs/plan/*·코드·.env.example 에 이미 답이 있으면 백지 질문 대신 "지금 X로 돼 있던데, 유지합니까?" 확인 질문으로 바꾼다. 이미 아는 걸 물으면 사용자는 인터뷰를 신뢰하지 않는다.

### 9. 산출물 생성
- 8단계가 끝나면 `references/templates.md` 의 형식대로 5개 파일을 `docs/plan/` 에 쓴다.
- 루트 `plan.md` 를 답변 내용으로 갱신한다 (블럭 체크리스트 = 3단계 MVP 순서). **블럭 3원칙**: 블럭 체크리스트의 각 블럭은 ① 눈에 보이는 결과 하나 ② 앞 블럭만 있으면 되고 뒤 블럭을 기다리지 않음 ③ 이 블럭에 필요한 것만(전체 DB를 첫 블럭에 만들지 않기).
- ⑦ 답에 나온 연동의 키가 `.env.example` 에 없으면 추가한다 (**`.env` 는 절대 건드리지 않는다**). 연동 목록엔 기본 스택(Supabase·Vercel)이 항상 들어가고, Cloudflare 는 7-7/7-8 판정에 해당할 때만 사유와 함께 들어간다. 외부 서비스는 전부 실명으로 적는다.
- 모순 검사: 화면에 있는데 저장 안 하는 데이터 / 저장하는데 아무도 못 보는 데이터 / 버튼의 목적지 없는 화면 — 발견하면 해당 단계로 돌아가 되묻는다.

### 9.5 허점 라운드 (따져보기)
산출물을 쓴 직후, 빌드로 넘어가기 전에 **검증 인터뷰 한 라운드**를 돈다. 만들 때의 질문이 아니라 완성본을 두들기는 질문이다.
- 문서 전체에서 허점 후보를 심각한 순으로 뽑는다: **빈 결정**(실패하면? 지우면 어떻게 되나? 누가 보나?) · **문서 간 모순**(위 모순 검사에서 걸린 것 포함) · **뜬구름 표현**("잘", "적절히", "나중에"). 특히 05-db.md 의 관계·권한·삭제정책에 구멍이 없는지 본다.
- **한 번에 하나씩**, AskUserQuestion 카드로 따져 묻는다 (추천 답 1순위 — 규칙 9). 답이 나올 때마다 해당 문서에 즉시 반영한다.
- **최대 5문.** 사용자가 "그만"이라 하면 즉시 멈춘다. 끝나면 바뀐 것과 남은 구멍(`[미정]`)을 3줄로 요약한다.
- 말투: 몰아세우지 않는다. 개업 전날 선배가 가게를 한 바퀴 돌며 "이건 어쩔 건데?" 묻는 톤. 여기서 아낀 10분이 나중에 10시간이다.

### 10. 빌드 연결
- 마지막에 말하라: "설계 끝. 이제 `plan.md` 의 1번 블럭부터 시작하자 — '1번 블럭 만들어줘' 라고 하면 된다."

## 금지
- 질문을 건너뛰고 그럴듯한 기획서를 지어내는 것 (**답 안 받은 칸은 전부 `[미정]`**).
- 산출물에 이모지 사용.
- MVP 4개 이상 (3개로 자를 때까지 3-C1 토너먼트 질문 반복).
- `.env` 읽기·쓰기.
